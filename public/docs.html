血液供應鏈網路 (Blood Supply Chain Network) - 系統交付文件

文件版本： 1.1.0 | 系統架構師： Google AI Studio (Senior HIS Architect)

本文件旨在提供給醫院業主、系統維運人員及使用者，包含詳細系統架構、操作手冊、邏輯檢驗規則及資料庫結構。

1. 系統功能架構 (System Functional Architecture)

本系統分為 供血端 (LIMS 模擬器) 與 需求端 (供應鏈與醫院) 兩大模組，透過單一介面模擬端到端 (End-to-End) 血液生命週期，確保從「手臂到手臂 (Vein-to-Vein)」的高度可追溯性與安全性。

  • 前端技術棧： React 18, Vite, TypeScript, Tailwind CSS, Lucide React
  • 後端技術棧： Node.js, Express.js (Port 3000)
  • 邊緣資料庫： SQLite (Better-SQLite3) 運行於本地系統，具備無縫離線能力與快取
  • 多語系架構 (i18n)： 系統內建繁體中文、英文、越南文切換，滿足跨國與移工操作落地需求。

2. 系統操作說明與檢驗邏輯 (Operation Manual & Validation Logic)

2.1 提供端：血站管理 / LIMS Simulator

負責模擬各捐血中心的作業流程與實驗室資訊系統 (LIMS)。

  • 【階段A】註冊與採血 (Donation)： 負責建立捐血者檔案(Donors)，登錄性別、年齡。建立問卷調查防禦線。
    [邏輯檢驗] 台灣身分證字號驗證： 嚴格檢查首字母對應的縣市代碼、第2碼的性別代碼(1或2)，以及透過「權重乘積和模數10 (Modulus 10)」驗證檢查碼，失敗一律阻攔，不允許建立假資料。
  • 【階段B】檢驗實驗室 (Lab Testing)： 收取試管，對血袋進行 ABO / RhD 驗明正身。進行傳染病 (IDM - Infectious Disease Marker) 檢測。只有 CLEARED (陰性) 才能往下流動；若為 REACTIVE (陽性) 則全批銷毀。
  • 【階段C】製造分離 (Processing)： 將全血透過離心技術，分離為 紅血球(RBC, E000100) 與 血小板(Platelets, E420600) 或 血漿(Plasma)。
  • 【階段D】品管核發 (QA Release / Cross-check)： 核發前，系統將血品條碼、血型與實驗室檢驗報告比對。若完全吻合，才賦予 ISBT-128 條碼 並放行至總部倉庫。

2.2 需求端：醫院與供應鏈管理 (Hospital & Supply Chain)

處理血庫調度、醫院下單、派車與臨床輸血，並結合 AI 計算。

  • (1) 下單 (Hospital Operator)： 醫院提出需求，填寫病人資訊、血型與優先層級 (Routine / ASAP / STAT)。系統將依據需求產生即時的 手術部位感染高危險 (HICI) 評分。
  • (2) AI 派送與調度 (Dispatcher)： 在庫存短缺或大量傷患 (MTP) 時，由 AI (模擬演算法) 接管調度規則，找出最短路徑及資源配置最佳解。
  • (3) 倉庫出庫 (Warehouse Issuer)： 庫管人員針對訂單掃描出庫，系統動態生成血袋條碼清單，並追蹤血品效期。
    [防呆設計] 效期監控機制： 出庫時，表格自動驗證血袋 expiryDate，如已過期將整行標示為紅底並打上 EXPIRED 與 WASTED 標籤，絕對禁止發配。
  • (4) 冷鏈追蹤 (Courier)： 運送途中監聽各階段的物流溫度，若超出正常範圍 (2~6度等)，護理師端將自動拒收。
  • (5) 床邊核對 (Bedside Verification - Nurse)： 臨床端進行三方條碼核對與病人驗明。這也是整套供應鏈的重點防護網。
    [防呆設計] ISBT 128 條碼解析規範：
    系統前台元件 (BedsideVerificationView) 強制限制：捐血碼必需符合 13 碼 ISBT 規範 =(13碼) 格式。(e.g., =W0000 24 10000101)。
    若格式有誤，會觸發前端 validateISBT128 失敗，按鈕強制反灰。

    [核心檢驗] ABO / RhD 相容性配對引擎：
    病患血型 O 型 → 只能接受 O 型。
    病患血型 A 型 → 可接受 A 型、O 型。
    病患血型 B 型 → 可接受 B 型、O 型。
    病患血型 AB 型 → 可接受 AB, A, B, O 型 (萬能受血者)。
    RhD 相容邏輯：陰性(-) 患者只能接受陰性(-) 的血。若為陽性患者，陰性/陽性皆可。
  • (6) 醫療主管覆審與 KPI (Manager / Auditor)： 大量輸血 (MTP)、未配血的緊急救援，將進入 Dual Review 覆審流程。

3. 資料庫實體關聯圖與結構 (Database Entity Relationship)

系統使用邊緣關聯式資料庫 (SQLite)，所有關聯嚴格參照。

資料表 (Table) 主鍵 (Primary Key) 重要欄位與關聯 (Fields & Foreign Keys) 設計說明與約束
donors (捐血者) id (TEXT) nationalId(UNIQUE), name, dob, bloodType, rhd 捐血者基本資料，身分證有 UNIQUE 唯一值約束
questionnaires (問卷紀錄) id (TEXT) donorId (FK), isPassed, answersJson 保存訪談歷史與適宜性
donations (採血紀錄) id (TEXT) donorId(FK), questionnaireId(FK), volume, type Id 會映射為 ISBT 全球唯一捐血碼
lab_tests (檢驗報告) id (TEXT) donationId (FK), abo, rhd, idmStatus 記錄三區化驗報告(CLEARED/REACTIVE)
components (分裝血庫存) id (TEXT) donationId(FK), productCode, status, expiryDate 最終被送至中央 Hub 的核發單位。包含過期計算。

4. 部署維運及交付注意事項

  1. 連線失效問題 (關於對外發布的分享網址無效)：
    Google AI Studio 的 Share (Shared App URL) 分享功能，需要等待發布流程自動把 dist/ 靜態檔案建立完畢，並掛載至 Cloud Run 後才會生效。若開啟分享網址遇到 "404 Page not found" 或是 "The requested URL was not found on this server"，通常是因為：
    (a) 剛剛結束改版，自動化 CI/CD pipeline 仍在佈署中，只需等待 1~3 分鐘重新整理即可。
    (b) 尚未主動按下右上角介面的 "Share" 更新按鈕，導致舊狀態失效。
  2. 本機與離線測試 (Offline Sync Simulator)：
    本系統為滿足醫院緊急網路中斷時的情境，特別設計了「Offline Mode (離線模式)」按鈕。開啟後前端 API 不會連線後台，將快取積壓於 Local Storage，復網後再發送 Sync 批次同步，開發者請利用右上角紅色 Offline 按鈕測試。
  3. 重新起始系統：
    您可以隨時於畫面上方點選「Reset System Data」按鈕，這會呼叫後端 API，將資料庫徹底清空並補入預設的 10~15 筆完美擬真教學資料，不會影響穩定性。
